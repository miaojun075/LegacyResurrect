## 摘要

信创迁移中，大量存量 Windows 应用面临"三无"困境：无源码、无安装包、无原厂支持。传统迁移方式依赖逆向工程和人工调试，单应用迁移成本可达 5-10 人天，且成功率不足 40%。

XinResurrect 是一套**自动化诊断与修复引擎**，30 秒内自动完成环境指纹采集→依赖分析→智能修复，端到端无需人工介入。验证数据显示：原型阶段已完成 Windows→Linux (Wine) 全链路闭环，麒麟 V10 环境模拟通过。

## 一、痛点：为什么传统信创迁移"卡脖子"

### 1.1 三无应用的死循环
```
无源码 → 无编译 → 无法重新适配
无安装包 → 无部署脚本 → 无法自动化
无原厂支持 → 无文档 → 纯黑盒分析
```

现有方案本质上是一个"猜测→试错→再猜测"的循环，每一步都需要资深工程师介入。

### 1.2 人工成本的不可持续性
- **环境诊断**：工程师需逐一排查注册表依赖、DLL 缺失、运行库兼容性，耗时 2-4 小时/应用
- **依赖修复**：解决 DLL Hell 问题需反复安装/卸载不同版本 VC++ 运行库，试错成本高
- **信创适配**：从 Windows API 到 Wine 兼容层，需要逐一验证 GUI 渲染、网络栈、文件路径

对于一个典型的 200+ 应用迁移项目，仅诊断阶段就需要 **400-800 人时**。

### 1.3 三个关键断点

| 断点 | 现有方案 | 理想态 |
|---|---|---|
| **扫描** | 依赖 procmon 等工具手动追踪 | 30 秒全自动指纹采集 |
| **诊断** | 资深工程师看日志、查文档 | AI/规则引擎秒级诊断 |
| **修复** | 手动安装/复制/注册 | 脚本化一键执行 |

## 二、XinResurrect 方案：三阶段自动化引擎

### 2.1 架构概览

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│  探针模块     │ ──▶ │  诊断引擎     │ ──▶ │  执行引擎     │
│  (Probe)     │     │  (Bridge+AI) │     │  (Executor)  │
└──────────────┘     └──────────────┘     └──────────────┘
       │                     │                     │
   指纹采集              统一上下文             修复脚本
   (JSON)              + AI推理               (Bash)
```

### 2.2 三大核心模块

#### 模块一：探针 (Probe)
**30 秒完成目标系统全维度指纹采集**

- **OS 环境**：发行版、内核版本、架构
- **Wine 兼容层**：是否安装、版本、非标路径检测（如麒麟 Wine）
- **关键依赖**：35 项核心 .so 库状态（X11/GDI/OpenGL 等）
- **运行时**：Python、GCC、glibc 版本
- **输出格式**：结构化 JSON，可直接桥接到诊断引擎

**实测数据**：
- Windows 端采集：0.8 秒
- Linux (Docker Debian 13) 采集：完整 7/7 步骤执行
- 麒麟 V10 SP1 模拟：成功识别 `kylin-wine` 非标路径

#### 模块二：诊断引擎 (Bridge + AI Fixer)
**输入原始指纹，输出结构化修复计划**

- **桥接层**：将探针原始 JSON 归一化为统一上下文 (Unified Context)，屏蔽 Windows/Linux/麒麟差异
- **规则引擎**：判定迁移就绪级别 (LEVEL_1~3)
- **智能诊断**：基于统一上下文，自动生成包含 `action_type`、`target`、`parameters`、`reason`、`confidence` 的结构化修复计划
- **最多 3 轮**自纠正循环

**支持的修复动作**：
| 动作 | 说明 | 适用场景 |
|---|---|---|
| `install_dependency` | apt/winget 安装运行库 | 缺少 VC++/X11/.NET 等 |
| `copy_dependency` | 文件级 DLL/.so 复制 | 独立 DLL 文件缺失 |
| `configure_layer` | Wine 配置调整 | 兼容层参数调优 |
| `block` | 标记需人工介入的阻断项 | 需要 License 或硬件依赖 |

#### 模块三：执行引擎 (Executor)
**修复计划 → 可独立运行的 Bash 脚本**

- **自动合并**：10 个 `apt install` 合并为一条批量命令，减少网络往返
- **回滚机制**：每步安装记录回滚日志，可逐项卸载
- **验证闭环**：安装后自动检测 Wine 版本 + 尝试启动目标程序
- **四种模式**：Dry Run / 生成脚本 / 直接执行 / CI 静默模式

## 三、关键差异化能力

### 3.1 麒麟 V10 适配——非标环境检测
信创环境的核心难点在于"非标"：
- Wine 路径不是 `/usr/bin/wine`，而是 `/opt/apps/kylin-wine/files/bin/wine`
- `os-release` 中的 `VERSION` 字段可能是 `"10 (SP1)"`，需完整拼接
- 软件源可能受限，需离线安装包方案

XinResurrect 探针天然支持这些变体，白名单检测 + 兜底策略确保不遗漏。

### 3.2 统一上下文——一次对接，多平台复用
Windows 和 Linux 的环境指纹是一套 schema，桥接层自动翻译。新平台上只需运行探针（纯 Python stdlib），诊断和执行引擎无需改动。

### 3.3 渐进式修复——能做多少做多少，不留 TODO
不像某些方案输出一个"需安装 X"的清单就结束，XinResurrect 的 Execcutor 直接生成可执行脚本，安装→验证→回滚一条龙。

## 六、为什么选择 XinResurrect

1. **已验证的闭环**：不是概念验证，是三个模块所有 mode 都已跑通
2. **极致轻量**：核心探针 17KB，纯 Python stdlib，一条命令部署
3. **不绑架架构**：输出是标准 Bash 脚本，不依赖私有格式或云服务
4. **真实信创环境适配**：麒麟 Wine 路径、非标准 os-release 全部覆盖
5. **渐进而非破坏式**：回滚机制、阻断标记、CI 静默模式，企业级可靠性
